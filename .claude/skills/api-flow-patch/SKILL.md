## 使用场景

用户需要：
- 补充 api-flow 分析时遗漏的业务流
- 手写业务流描述后自动转化为标准格式
- 直接口述业务流程，自动补充到 api_flow.json

### 典型触发方式

1. **文件触发**：用户在 `api_parser/api_flow.md` 中手动编写了业务流
2. **口述触发**：用户直接描述业务流，如 "补充一个接口CRUD的流程：登录→创建项目→创建接口→删除接口"
3. **混合触发**：用户同时提供文件和口述补充

## 核心规则

### 1. 输入与输出
- **输入来源（按优先级）：**
  1. 用户直接口述的业务流描述
  2. `api_parser/api_flow.md` 文件（用户手写的业务流清单）
- **依赖文件：**
  - `api_parser/api-spec.json` — 接口定义（用于匹配接口和生成步骤）
  - `api_flow/api_flow.json` — 已有业务流（用于去重判断）
- **输出：** 直接更新 `api_flow/api_flow.json`（新增流程追加到 business_flows 或 edge_cases）

### 2. 补充流程

#### Step 1: 读取已有数据
- 读取 `api_flow/api_flow.json`，提取已有流程 ID 列表
- 读取 `api_parser/api-spec.json`，提取所有接口定义

#### Step 2: 解析用户输入
从 MD 文件或用户描述中提取业务流：

**MD 文件解析规则：**
- 以 `###` 开头的行是流程标题
- 标题格式 `数字、流程名：接口调用关系：步骤1--->步骤2--->步骤3`
- 每个步骤是中文描述（如"用户登陆"、"创建项目"、"查询接口列表"）
- 去除已有流程中已覆盖的（通过步骤语义比对）

**口述解析规则：**
- 用 `→` 或 `->` 或 `然后` 或 `之后` 分隔步骤
- 每个步骤是接口操作的描述

#### Step 3: 步骤→接口匹配
对每个步骤描述，匹配到 api-spec.json 中的具体接口：

**匹配策略（按优先级）：**
1. **精确匹配**：步骤描述包含接口 `name` 字段
2. **路径关键词**：步骤描述包含路径关键词（如"注册"→ /api/users/register）
3. **语义匹配**：根据上下文推断（如"登陆"→ POST /api/users/login）
4. **模块+操作**：根据模块名+操作类型推断（如"创建接口"+"接口模块"→ 对应POST接口）

匹配时需要同时确定：
- HTTP 方法（创建=POST，查询=GET，修改=PUT，删除=DELETE）
- 具体路径（从 api-spec.json 中查找）
- 是否需要认证（从 auth 字段获取）
- 请求参数和请求体（从接口定义中获取）

#### Step 4: 去重判断
将解析出的流程与已有流程比对：

**判断为"已存在"的条件（满足任一）：**
1. 流程包含的接口集合与已有流程完全相同
2. 流程是已有流程的子集
3. 已有流程是该流程的超集

**判断为"需要补充"的条件：**
1. 包含已有流程中没有的新接口组合
2. 步骤顺序不同（代表不同的测试意图）
3. 包含新的异常校验步骤

#### Step 5: 生成标准流程
对需要补充的流程，按 api-flow 的输出格式生成：

1. 生成流程 ID（英文小写+下划线，如 `interface_crud`）
2. 自动编排 input_vars / output_vars（步骤间数据传递）
3. 添加必要的认证前置步骤
4. 末尾添加数据清理步骤
5. 生成 request_example（从接口定义中提取字段和示例值）

#### Step 6: 更新文件
1. 将新流程追加到 `api_flow.json` 的 `business_flows` 或 `edge_cases` 数组
2. 更新 `summary` 中的统计数字
3. 重新计算覆盖率
4. 保存文件

#### Step 7: 展示结果
向用户展示：
- 新增了多少条业务流
- 每条新增流程的概要
- 更新后的覆盖率

### 3. MD 文件位置

默认读取：`output/{项目名}/api_parser/api_flow.md`

如果用户指定了其他路径，以用户指定的为准。

### 4. 处理原则

- **不删除**已有流程，只追加新流程
- **不修改**已有流程的步骤内容
- 如果用户描述的流程与已有流程高度相似，先询问用户是否需要替换
- 无法匹配到具体接口的步骤，标记为 `note` 而非跳过
- 一个 MD 文件可能包含多条流程，逐条处理

### 5. ID 生成规则

- 基于流程名称生成英文 ID
- 已有 ID 不重复使用
- 格式：`{模块}_{操作}`（如 `interface_crud`、`batch_delete`）
