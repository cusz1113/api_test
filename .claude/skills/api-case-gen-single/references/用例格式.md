# 单接口用例输出格式定义

## 单接口用例文件结构

每个接口一个 JSON 文件，命名规则：`{method}_{path去掉前导斜杠并将/替换为_}.json`

### 顶层结构

```json
{
  "target_endpoint": { ... },
  "setup_steps": [ ... ],
  "test_steps": [ ... ],
  "generated_at": "2026-04-25T20:30:00+08:00",
  "source": "api_parser/api-spec.json"
}
```

---

## target_endpoint — 目标接口信息

```json
{
  "endpoint_id": "register_api_users_register_post",
  "method": "POST",
  "path": "/api/users/register",
  "name": "用户注册",
  "description": "注册新用户",
  "tags": ["用户管理"]
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| endpoint_id | ✅ | api-spec.json 中接口的唯一标识 |
| method | ✅ | HTTP 方法 |
| path | ✅ | 接口路径 |
| name | ✅ | 接口名称 |
| description | ❌ | 接口描述 |
| tags | ❌ | 接口标签 |

---

## setup_steps — 前置依赖步骤

与业务流用例的步骤格式一致（见 api-case-gen 的用例格式），额外增加以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| ref | ❌ | 引用来源，格式 `"{文件名}.setup_steps[{步骤号}]"` |
| note | ❌ | 备注说明，标注该前置步骤的用途 |

**公共前置复用示例：**
```json
{
  "step_no": 2,
  "name": "登录获取token",
  "ref": "POST_api_users_login.json.setup_steps[2]",
  "note": "公共前置：获取认证token",
  "method": "POST",
  "url": "{{base_url}}/api/users/login",
  ...
}
```

### setup_steps 可以为空数组

```json
"setup_steps": []
```

表示该接口无前置依赖（如注册接口本身不需要 token）。

---

## test_steps — 测试场景步骤

在标准步骤格式基础上，增加以下专有字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| scenario | ✅ | 场景类型标识 |
| missing_field | ❌ | 缺少的字段名（scenario=missing_required） |
| inject_field | ❌ | 注入的字段名（scenario=sql_injection/xss） |
| boundary_type | ❌ | 边界类型（scenario=boundary）：empty_string/max_length/min_value/max_value/zero/negative |

### scenario 类型完整列表

| scenario 值 | 说明 | 典型 status_code |
|------------|------|-----------------|
| `positive` | 正向用例 | 200/201 |
| `missing_required` | 缺少必填字段 | 422 |
| `wrong_type` | 参数类型错误 | 422 |
| `format_error` | 格式校验失败 | 422 |
| `boundary` | 边界值测试 | 422 |
| `unique_conflict` | 唯一约束冲突 | 422/409 |
| `not_found` | 资源不存在 | 404 |
| `unauthorized` | 未认证访问 | 401 |
| `forbidden` | 无权限访问 | 403 |
| `sql_injection` | SQL注入测试 | 400/422/500 |
| `xss` | XSS注入测试 | 200（防护成功）或 400 |

### 各场景 body 构造规则

#### positive（正向）
```json
{
  "body": {
    "username": "{{random_username}}",
    "password": "Test@123456",
    "password_confirm": "Test@123456",
    "email": "{{random_email}}",
    "mobile": "{{random_mobile}}"
  }
}
```

#### missing_required
```json
{
  "scenario": "missing_required",
  "missing_field": "username",
  "body": {
    "password": "Test@123456",
    "password_confirm": "Test@123456",
    "email": "{{random_email}}",
    "mobile": "{{random_mobile}}"
  }
}
```

#### wrong_type
```json
{
  "scenario": "wrong_type",
  "inject_field": "age",
  "body": {
    "username": "{{random_username}}",
    "age": "不是数字"
  }
}
```

#### format_error
```json
{
  "scenario": "format_error",
  "inject_field": "email",
  "body": {
    "username": "{{random_username}}",
    "email": "不是邮箱"
  }
}
```

#### boundary
```json
{
  "scenario": "boundary",
  "boundary_type": "empty_string",
  "inject_field": "username",
  "body": {
    "username": "",
    "password": "Test@123456"
  }
}
```

#### unique_conflict
```json
{
  "scenario": "unique_conflict",
  "body": {
    "username": "{{1.extract.username}}"
  }
}
```
使用 setup_steps 中已创建的重复数据。

#### not_found
```json
{
  "scenario": "not_found",
  "params": {
    "project_id": 999999
  }
}
```

#### unauthorized
```json
{
  "scenario": "unauthorized",
  "headers": {
    "Content-Type": "application/json"
  }
}
```
不带 Authorization header。

#### sql_injection
```json
{
  "scenario": "sql_injection",
  "inject_field": "username",
  "body": {
    "username": "1' OR 1=1--",
    "password": "Test@123456"
  }
}
```

#### xss
```json
{
  "scenario": "xss",
  "inject_field": "username",
  "body": {
    "username": "<script>alert(1)</script>",
    "password": "Test@123456"
  }
}
```

    "total_cases_files": 61,
    "total_test_steps": 385,
    "by_scenario": {

### common_setups — 公共前置清单

记录被多个接口共享的前置链路，方便执行器复用。

## 变量体系

与 api-case-gen 完全一致：

| 变量 | 说明 | 示例值 |
|------|------|--------|
| `{{base_url}}` | API 基础地址 | `http://124.222.144.182:8888` |
| `{{random_username}}` | 随机用户名（8位） | `tsta3f2k9x` |
| `{{random_email}}` | 随机邮箱 | `tsta3f2k9x@test.com` |
| `{{random_mobile}}` | 随机手机号 | `13812340001` |
| `{{random_string_4}}` | 4位随机字符串 | `x7k2` |
| `{{random_string_8}}` | 8位随机字符串 | `x7k2m9p4` |
| `{{N.extract.xxx}}` | 引用第 N 步提取的变量 | `{{2.extract.token}}` |

## 质量检测规则

与 api-case-gen 三维度检测一致：

### 维度一：数据引用错误
- setup_steps + test_steps 中所有变量引用的步骤号 < 当前步骤号
- 引用变量在目标步骤 extract 中存在
- extract 路径在 response.fields 中有对应字段

### 维度二：断言规则正确性
- 正向用例 status_code 与 api-spec response.status_code 一致
- 异常用例 status_code 为合理错误码
- body_fields 字段名在 response.fields 中存在
- 204 无 body_fields

### 维度三：编造数据检测
- body 无文档外字段
- 正向用例 body 包含所有 required 字段
- 字段值类型正确
- Content-Type 与接口一致
- headers 无文档外 header
- params 无文档外参数
- extract 路径无编造
