# 测试用例输出格式定义

## 顶层结构

```json
{
  "project": "web-test-platform",
  "generated_at": "2026-04-22T22:00:00+08:00",
  "source": "api_flow/api_flow.json",
  "base_url": "http://124.222.144.182:8888",
  "summary": {
    "total_flows": 13,
    "total_cases": 13,
    "by_priority": {
      "P0": 10,
      "P1": 3
    }
  },
  "cases": [ ... ]
}
```

### summary 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| total_flows | ✅ | 业务流总数 |
| total_cases | ✅ | 用例总数（= 业务流数，一个流程一条正向用例） |
| by_priority | ✅ | 按优先级统计用例数 |

---

## cases — 用例列表

### 用例结构

```json
{
  "id": "user_auth_001",
  "name": "用户注册登录认证流程-正向",
  "flow_id": "user_auth",
  "flow_name": "用户注册登录认证流程",
  "priority": "P0",
  "tags": ["用户管理", "正向"],
  "preconditions": ["测试环境可用", "无同名用户存在"],
  "steps": [ ... ],
  "extract_vars": {
    "username": "{{1.body.username}}",
    "token": "{{2.body.token}}"
  }
}
```

### 用例字段说明

| 字段 | 必填 | 类型 | 说明 |
|------|------|------|------|
| id | ✅ | string | 用例唯一标识，格式 `{flow_id}_{序号3位}` |
| name | ✅ | string | 用例名称，格式 `{业务流名称}-正向` |
| flow_id | ✅ | string | 关联的业务流ID |
| flow_name | ✅ | string | 关联的业务流名称 |
| priority | ✅ | string | 优先级，继承业务流优先级 |
| tags | ✅ | string[] | 标签（接口模块标签 + "正向"） |
| preconditions | ✅ | string[] | 前置条件列表 |
| steps | ✅ | array | 测试步骤列表 |
| extract_vars | ✅ | object | 用例级变量提取汇总（所有步骤的提取变量） |

---

## steps — 测试步骤

### 步骤结构

```json
{
  "step_no": 1,
  "name": "注册",
  "method": "POST",
  "url": "{{base_url}}/api/users/register",
  "headers": {
    "Content-Type": "application/json"
  },
  "params": {},
  "body": {
    "username": "{{random_username}}",
    "password": "Test@123456",
    "password_confirm": "Test@123456",
    "email": "{{random_email}}",
    "mobile": "{{random_mobile}}"
  },
  "extract": {
    "username": "$.username"
  },
  "assert": {
    "status_code": 200,
    "body_fields": {
      "id": "not_null",
      "username": "not_null"
    }
  }
}
```

### 步骤字段说明

| 字段 | 必填 | 类型 | 说明 |
|------|------|------|------|
| step_no | ✅ | integer | 步骤序号（从1开始） |
| name | ✅ | string | 步骤名称（继承业务流步骤） |
| method | ✅ | string | HTTP 方法 |
| url | ✅ | string | 完整请求URL，`{{base_url}}` + path |
| headers | ✅ | object | 请求头 |
| params | ✅ | object | 查询参数 |
| body | ❌ | object/array | 请求体 |
| extract | ✅ | object | 从响应中提取的变量（key=变量名, value=JSONPath） |
| assert | ✅ | object | 断言规则 |

---

### headers 生成规则

```json
{
  "Content-Type": "application/json",
  "Authorization": "Bearer {{2.extract.token}}"
}
```

| 条件 | 添加的 Header |
|------|--------------|
| 有 request_body | `Content-Type: application/json` |
| requires_auth == true | `Authorization: Bearer {{N.extract.token}}` |

---

### extract — 变量提取

从响应体中提取后续步骤需要的变量。

| 提取场景 | JSONPath | 说明 |
|----------|----------|------|
| 创建资源 | `$.id` | 提取新建资源的ID |
| 登录成功 | `$.token` | 提取认证Token |
| 刷新Token | `$.token` | 提取新Token |
| 注册返回 | `$.username` | 提取实际用户名 |
| 注册密码 | 固定值 | 密码不在响应中，用请求中的值 |

**变量引用格式：** `{{N.extract.var}}` 表示引用第N步提取的变量

**示例：**
```json
{
  "extract": {
    "project_id": "$.id"
  }
}
```
后续步骤中引用：`{{2.extract.project_id}}`

### assert — 断言规则

```json
{
  "status_code": 200,
  "body_fields": {
    "id": "not_null",
    "username": "not_null",
    "token": "not_null"
  }
}
```

#### status_code 规则

**直接从业务流步骤的 `response.status_code` 读取，不做任何推测。**

- 步骤写 200 → 断言 200
- 步骤写 201 → 断言 201
- 步骤写 204 → 断言 204
- status_code 为 204 时，不生成 body_fields 断言

#### body_fields 规则

| 断言值 | 含义 |
|--------|------|
| `"not_null"` | 字段存在且不为 null/空 |
| `"type:integer"` | 字段类型为整数 |
| `"type:string"` | 字段类型为字符串 |
| `"type:array"` | 字段类型为数组 |
| `"eq:value"` | 字段等于指定值 |

**按接口类型的默认断言：**

| 接口类型 | 必须断言的字段 |
|----------|--------------|
| POST 创建 | `id`: not_null |
| POST 登录 | `token`: not_null |
| GET 列表 | 列表本身: not_null (数组) |
| GET 详情 | `id`: not_null |
| PUT 修改 | `id`: not_null |
| DELETE | 仅断言 status_code |

## extract_vars — 用例级变量汇总

将所有步骤的 extract 合并为一个对象，方便执行引擎统一管理：

```json
{
  "extract_vars": {
    "username": "{{1.extract.username}}",
    "password": "Test@123456",
    "token": "{{2.extract.token}}",
    "new_token": "{{4.extract.token}}"
  }
}
```

- 从 extract 中提取的变量：`{{N.extract.var}}`
- 从请求体中固定传入的变量（如密码）：直接使用字面值

## 变量体系

### 全局变量

| 变量 | 说明 | 示例值 |
|------|------|--------|
| `{{base_url}}` | API基础地址 | `http://124.222.144.182:8888` |
| `{{random_username}}` | 随机用户名 | `testuser_a3f2` |
| `{{random_email}}` | 随机邮箱 | `test_a3f2@example.com` |
| `{{random_mobile}}` | 随机手机号 | `13812340001` |
| `{{random_string_4}}` | 4位随机字符串 | `x7k2` |

### 步骤变量

| 变量 | 说明 | 引用格式 |
|------|------|----------|
| 提取变量 | 从响应中提取 | `{{N.extract.var}}` |
| 请求变量 | 从请求中传递 | `{{N.body.field}}` |

### 变量解析顺序

1. 全局变量替换（random_*）
2. 用例级变量替换（extract_vars）
3. 步骤间变量替换（`{{N.extract.var}}`）

---

## 数据质量规则

1. **步骤完整**：用例的步骤必须与业务流步骤一一对应
2. **变量连续**：extract 中的变量必须在后续步骤中被引用（或为最终产出）
3. **断言合理**：每个步骤必须有 status_code 断言
4. **无死变量**：extract_vars 中不应该有未被任何步骤引用的变量
5. **url 完整**：所有 url 必须包含 {{base_url}} 前缀
