# 业务流分析策略与模式库

## 分析流程

### Phase 1: 逐接口分析

对 api-spec.json 中的每个接口独立分析，提取以下信息：

#### 1.1 认证需求
- `auth.type == "bearer"` → 该接口需要登录前置步骤
- `auth.type == "none"` → 公开接口，可直接调用
- 需要认证的接口自动继承登录步骤

#### 1.2 资源依赖（外键识别）

通过请求体字段名识别外键依赖：

| 字段名 | 依赖资源 | 对应创建接口 |
|--------|----------|-------------|
| project_id | 项目 | POST /api/pro/projects |
| modules_id | 模块 | POST /api/pro/modules |
| env_id | 环境 | POST /api/pro/envs |
| suite_id | 套件 | POST /api/test/suites |
| cases_id | 用例 | POST /api/test/cases |
| task_id | 任务 | POST /api/plan/tasks |
| device_id | 设备 | POST /api/node/devices |

识别规则：
- POST/PUT 请求体中出现上述字段 → 需要前置创建对应资源
- 路径参数中出现 `{id}` → 需要前置获取资源 ID

#### 1.3 产出资源

根据接口方法和语义判断产出：

| 方法 + 语义 | 产出变量 | 说明 |
|-------------|----------|------|
| POST 注册 | username, password | 注册后的凭证 |
| POST 登录 | token | 认证令牌 |
| POST 刷新token | new_token | 新令牌 |
| POST 创建 | {resource}_id | 新资源的ID |
| POST 复制 | copied_id | 复制后的资源ID |

#### 1.4 约束识别

**唯一约束**（POST 创建时需要校验重复）：
- username、email、mobile（用户相关）
- name（项目、模块、环境等，需确认后端是否唯一）

**格式约束**（需要格式错误测试）：
- format == "email" → 邮箱格式校验
- format == "mobile" → 手机号格式校验
- pattern → 正则校验
- maxLength / minLength → 长度边界

**一致性约束**：
- password + password_confirm 同时存在 → 密码一致性校验

#### 1.5 测试关注点

根据接口特征自动生成测试关注点：

| 特征 | 关注点 |
|------|--------|
| auth.type == "bearer" | 未认证访问拒绝 |
| 路径参数 {id} | 不存在的ID |
| POST 创建 | 重复创建 |
| format 字段 | 格式错误 |
| password + confirm | 密码不一致 |
| 分页参数 | 边界值（-1, 999999） |
| PUT/PATCH | 数据更新验证 |
| DELETE | 资源删除验证 |
| 登录接口 | 密码错误、用户不存在 |
| 复制接口 | 不存在的源ID |

### Phase 2: 依赖图谱

#### 2.1 资源层级关系

```
项目 (project)
├── 模块 (module)
│   └── 套件 (suite)
│       └── 套件用例关联 (suite_case)
│           └── 用例 (case)
├── 环境 (env)
├── 任务 (task)
│   └── 任务套件关联 (task_suite)
└── 定时任务 (cron)
    └── → 引用任务
```

#### 2.2 依赖传播规则

- 创建子资源 → 必须先创建父资源
- 删除父资源 → 需要验证子资源是否级联删除
- 查询子资源 → 需要父资源的 ID

### Phase 3: 业务流串联模式

#### 3.1 CRUD 完整流程模式

适用于所有资源实体（项目、模块、环境、套件、用例等）：

```
登录（如需认证）
  ↓
创建资源 → 输出 resource_id
  ↓
查询列表 → 验证列表包含新创建的资源
  ↓
查询详情 → 验证详情数据正确
  ↓
修改资源 → 验证修改生效
  ↓
删除资源 → 验证删除成功
  ↓
查询详情 → 验证已不存在（可选）
  ↓
清理前置资源
```

#### 3.2 关联管理流程模式

适用于多对多关系（套件↔用例、任务↔套件）：

```
准备双方资源
  ↓
添加关联 → 输出 relation_id
  ↓
查询关联列表 → 验证关联存在
  ↓
修改关联属性（排序、跳过等）
  ↓
删除关联 → 验证删除成功
  ↓
清理资源
```

#### 3.3 执行+结果流程模式

适用于有执行结果的接口（运行用例/套件/任务）：

```
准备执行数据（套件+用例+任务）
  ↓
执行用例 → 触发单用例运行
  ↓
执行套件 → 触发套件运行
  ↓
执行任务 → 触发任务运行
  ↓
查询运行记录列表 → 验证记录存在
  ↓
查询运行记录详情 → 验证结果数据
  ↓
清理资源
```

#### 3.4 认证流程模式

```
注册 → 输出 username, password
  ↓
登录 → 输出 token
  ↓
校验 token → 验证 token 有效
  ↓
刷新 token → 输出 new_token
  ↓
校验新 token → 验证刷新生效
```

#### 3.5 公共步骤抽取

以下步骤在多个流程中重复使用，应抽取为公共前缀：

- **登录**：POST /api/users/login → token
- **创建项目**：POST /api/pro/projects → project_id
- **创建模块**：POST /api/pro/modules → module_id
- **创建套件**：POST /api/test/suites → suite_id

### Phase 4: 异常场景模式

#### 4.1 鉴权异常（auth）

| 场景 | 方法 | 说明 |
|------|------|------|
| 未认证访问 | GET 任意需认证接口 | 不携带 token |
| 无效 Token | GET 任意需认证接口 | 携带无效 token |
| Token 过期 | GET 任意需认证接口 | 携带已过期 token |
| 密码错误 | POST /login | 正确用户名+错误密码 |
| 用户不存在 | POST /login | 不存在的用户名 |

#### 4.2 边界值异常（boundary）

| 场景 | 说明 |
|------|------|
| 不存在的 ID | 路径参数传入 999999 等不存在的值 |
| 缺少必填字段 | POST 请求不传必填 body 字段 |
| 超长字符串 | 超过 maxLength 的字符串 |
| 格式错误 | email、mobile 等格式不合法 |
| 密码不一致 | password ≠ password_confirm |
| 分页异常 | page=-1, size=999999, page=0 |

#### 4.3 唯一约束异常（duplicate）

| 场景 | 说明 |
|------|------|
| 重复用户名 | 用已存在的 username 注册 |
| 重复项目名 | 用已存在的 name 创建项目 |
| 重复邮箱 | 用已存在的 email 注册 |

#### 4.4 级联验证（cascade）

| 场景 | 说明 |
|------|------|
| 删除项目→模块 | 删除项目后查询模块，期望404 |
| 删除模块→套件 | 删除模块后查询套件，期望404 |

#### 4.5 安全测试（security）

| 场景 | 说明 |
|------|------|
| SQL 注入 | 在字符串字段注入 SQL 语句 |
| XSS 注入 | 在字符串字段注入 HTML/JS 代码 |

## 覆盖率校验

### 校验方法

1. 收集 api-spec.json 中所有接口的 `method + path`（去尾部斜杠）
2. 遍历所有 business_flows 和 edge_cases 的 steps
3. 匹配 `method + path`（去尾部斜杠）
4. 计算覆盖率 = 已覆盖数 / 总数
5. 未覆盖的接口必须补充到流程或异常场景中

### 常见遗漏点

- POST 和 GET 同路径但不同方法（如 POST /api/plan/tasks 和 GET /api/plan/tasks）
- 路径尾部斜杠不一致（规范化处理）
- 详情查询接口（GET /xxx/{id}）容易遗漏
- 记录详情接口（GET /run/xxx/records/{id}）
