# 业务流输出格式定义

## 顶层结构

```json
{
  "project": "web-test-platform",
  "analyzed_at": "2026-04-22T21:30:00+08:00",
  "source": "api_parser/api-spec.json",
  "summary": {
    "total_endpoints": 61,
    "total_business_flows": 13,
    "total_edge_cases": 19,
    "coverage": "61/61",
    "coverage_pct": "100%",
    "uncovered": []
  },
  "business_flows": [ ... ],
  "edge_cases": [ ... ]
}
```

### summary 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| total_endpoints | ✅ | 源接口总数（来自 api-spec.json） |
| total_business_flows | ✅ | 业务流程总数 |
| total_edge_cases | ✅ | 异常场景总数 |
| coverage | ✅ | 覆盖率，格式 "已覆盖/总数" |
| coverage_pct | ✅ | 覆盖率百分比 |
| uncovered | ✅ | 未覆盖的接口列表（应为空数组） |

---

## business_flows — 业务流程

```json
{
  "id": "project_crud",
  "name": "项目完整CRUD流程",
  "priority": "P0",
  "description": "登录→创建→列表→详情→修改→删除",
  "steps": [ ... ]
}
```

### 流程字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| id | ✅ | 流程唯一标识，英文小写+下划线 |
| name | ✅ | 流程中文名称 |
| priority | ✅ | 优先级：P0 / P1 / P2 |
| description | ✅ | 流程描述（步骤概要） |
| steps | ✅ | 步骤列表（有序数组） |

### steps — 步骤格式

```json
{
  "name": "登录",
  "method": "POST",
  "path": "/api/users/login",
  "requires_auth": false,
  "input_vars": {"username": "1.username", "password": "1.password"},
  "output_vars": ["token"],
  "request_example": {"username": "testuser", "password": "Test@123456"},
  "params": {"page": 1, "size": 10},
  "note": "可选备注说明"
}
```

| 字段 | 必填 | 类型 | 说明 |
|------|------|------|------|
| name | ✅ | string | 步骤名称（从路径最后一段提取或自定义） |
| method | ✅ | string | HTTP 方法：GET / POST / PUT / DELETE / PATCH |
| path | ✅ | string | 接口路径（不带尾部斜杠） |
| requires_auth | ✅ | boolean | 是否需要认证 |
| input_vars | ❌ | object | 输入变量映射，key 为参数名，value 为 `"步骤号.变量名"` 格式 |
| output_vars | ❌ | string[] | 本步骤产出的变量名列表 |
| request_example | ❌ | object/array | 请求体示例，可含 `{{变量}}` 占位符 |
| params | ❌ | object | 查询参数示例 |
| note | ❌ | string | 备注说明（如期望状态码、特殊校验点） |

### input_vars 变量引用格式

- `"1.token"` — 引用第 1 步产出的 token
- `"2.project_id"` — 引用第 2 步产出的 project_id
- `"4.suite_id"` — 引用第 4 步产出的 suite_id
- 字符串值可拼接变量：`"Bearer {1.token}"`

### output_vars 常见产出

| 变量名 | 含义 | 典型来源接口 |
|--------|------|-------------|
| token | 认证令牌 | POST /api/users/login |
| new_token | 刷新后的令牌 | POST /api/users/refresh |
| project_id | 项目ID | POST /api/pro/projects |
| module_id | 模块ID | POST /api/pro/modules |
| env_id | 环境ID | POST /api/pro/envs |
| suite_id | 套件ID | POST /api/test/suites |
| case_id | 用例ID | POST /api/test/cases |
| task_id | 任务ID | POST /api/plan/tasks |
| cron_id | 定时任务ID | POST /api/cron/crontab |
| device_id | 设备ID | POST /api/node/devices |
| time_id | 设备时间标识 | POST /api/node/devices |
| rel_id | 关联ID | POST /api/test/suites/{id}/cases |

### 占位符约定

| 占位符 | 说明 | 用途 |
|--------|------|------|
| `{{random_string_4}}` | 4位随机字符串 | 项目名、套件名去重 |
| `{{random_username}}` | 随机用户名 | 注册时避免冲突 |
| `{{random_email}}` | 随机邮箱 | 注册时避免冲突 |
| `{{random_mobile}}` | 随机手机号 | 注册时避免冲突 |
| `{{global.test_username}}` | 全局测试账号名 | 异常场景引用已存在账号 |
| `{{global.test_password}}` | 全局测试账号密码 | 异常场景登录 |

## edge_cases — 异常场景

```json
{
  "id": "no_auth",
  "name": "未认证访问项目接口",
  "priority": "P1",
  "category": "auth",
  "depends_on": null,
  "steps": [
    {
      "name": "projects",
      "method": "GET",
      "path": "/api/pro/projects",
      "note": "不携带token，期望401"
    }
  ]
}
```

### 异常场景字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| id | ✅ | 场景唯一标识 |
| name | ✅ | 场景中文名称 |
| priority | ✅ | 优先级：P0 / P1 / P2 |
| category | ✅ | 分类：auth / boundary / duplicate / cascade / security |
| depends_on | ❌ | 依赖的业务流程ID（null 表示无依赖） |
| steps | ✅ | 步骤列表（格式与业务流步骤一致） |

### category 分类说明

| 分类 | 说明 | 典型场景 |
|------|------|----------|
| auth | 鉴权异常 | 未认证、无效Token、Token过期 |
| boundary | 边界值 | 不存在的ID、超长字符串、负数、超大分页 |
| duplicate | 唯一约束 | 重复用户名、重复项目名 |
| cascade | 级联影响 | 删除父资源后子资源验证 |
| security | 安全测试 | SQL注入、XSS注入 |

---

## 数据质量规则

1. **100%覆盖**：所有接口必须被至少一个流程或异常场景覆盖
2. **数据清理**：每条业务流程末尾必须删除创建的测试数据
3. **变量连续**：input_vars 引用的步骤号必须 < 当前步骤号
4. **路径一致**：步骤中的 path 必须与 api-spec.json 中的接口路径完全一致（去尾部斜杠）
5. **无循环依赖**：业务流之间可以有依赖，但不能循环
